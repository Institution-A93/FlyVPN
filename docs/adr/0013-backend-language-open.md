# ADR-0013: Язык backend-сервисов — Go

- Статус: accepted
- Дата: 2026-06-09 (выбор зафиксирован), пересмотрен из `proposed`

## Контекст
Язык backend-сервисов (`config-api`, `orchestrator`) изначально стоял в спеке как «Go»,
был справедливо снят как requirement-галлюцинация и помечен открытым. После непредвзятого
взвешивания по критериям ниже выбран осознанно.

## Решение
Backend-сервисы пишем на **Go**. Обоснование (а не инерция спеки):
1. **Деплой на cattle**: `go build` → один статический бинарь, без рантайма/venv/node_modules
   на узле. Ложится на инвариант воспроизводимости узла.
2. **Малое, аудируемое дерево зависимостей** — важно для security-чувствительного продукта;
   stdlib закрывает HTTP/HMAC/XML, плюс `golang.org/x/crypto/md4` (NT-hash) и `pgx`.
3. **orchestrator** — низкоуровневые health-пробы (UDP/TLS), запуск tofu/ansible/SSH,
   конкурентность через горутины: родная стихия Go.
4. Экосистема вокруг (Xray-core и пр.) — Go.
5. Достаточная скорость разработки для вебхука + оркестратора.

Рассмотренные альтернативы: TypeScript (Bun) — быстрый config-api, но деплой single-binary
и npm-supply-chain слабее, orchestrator менее удобен; Rust — избыточная церемония для I/O-bound
MMVP; Python — самый грязный деплой на cattle.

## Следствия
- Тулчейн Go; один язык на оба сервиса.
- Раннер миграций можно взять Go-нативный (golang-migrate/goose) либо оставить psql —
  решается при реализации.
