# ADR-0014: Хранение пароля при EAP-MSCHAPv2 — NT-hash

- Статус: accepted
- Дата: 2026-06-09

## Контекст
README §2.7 (шаг 5) предписывал bcrypt; ADR-0005 выбрал EAP-MSCHAPv2. Это
криптографически несовместимо: MSCHAPv2 (challenge-response) требует, чтобы сервер имел
**NT-hash** (MD4 от UTF-16LE пароля) либо cleartext; bcrypt односторонний — из него
NT-hash не получить, а открытый пароль клиент в MSCHAPv2 не присылает.

Канал доставки (`.mobileconfig` для IKEv2 VPN) подтверждённо ограничивает парольный EAP
**только EAP-MSCHAPv2** (Apple/strongSwan: иные парольные EAP → IKE_AUTH failure).
Сертификатный путь — EAP-TLS. EAP-TTLS/PAP (который позволил бы bcrypt) Apple для IKEv2
VPN не поддерживает. Источник: docs.strongswan.org/.../appleIkev2Profile.

## Решение
Хранить **NT-hash** (вариант A). В `auth_credentials` колонка переименована
`password_hash` → `nt_hash` (hex MD4, 32 hex-символа). FreeRADIUS (`rlm_sql` + `mschap`)
отдаёт его как `NT-Password`. bcrypt из схемы убран.

Митигация слабости NT-hash (несолёный MD4, быстрый брутфорс при утечке БД): пароль
генерим мы — случайный, высокой энтропии (≥32 символа); БД под FDE/at-rest шифрованием.
Усиление до «NT-hash, зашифрованный at-rest» — возможный последующий шаг.

## EAP-TLS — отложено
EAP-TLS («без паролей») сильнее, но стоит PKI/CRL, перевыпуска сертов при продлении
(переустановка профиля) и заметно большего объёма работ. Отложено как осознанный апгрейд;
канал поддерживает оба метода, цена миграции = перевыпуск кредов + новые профили (что при
продлении и так происходит). Будет оформлено отдельным ADR при реализации.

## Следствия
- Миграция: колонка `nt_hash` вместо `password_hash`.
- README §2.7 шаг 5 и §4 скорректированы (NT-hash вместо bcrypt).
- Разблокирована FreeRADIUS-роль control-plane (rlm_sql + mschap, NT-Password).
- Авторизация по активной подписке (revoked_at / срок) — на стороне config-api/orchestrator.
