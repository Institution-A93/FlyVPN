# ADR-0014: Хранение пароля при EAP-MSCHAPv2 (конфликт с bcrypt)

- Статус: proposed
- Дата: 2026-06-09

## Контекст
README §2.7 (шаг 5) предписывает хранить пароль как **bcrypt**. ADR-0005 выбрал
**EAP-MSCHAPv2**. Это криптографически несовместимо:

- MSCHAPv2 — challenge-response. Сервер (FreeRADIUS, модуль `mschap`) обязан вычислить
  ожидаемый ответ из **NT-hash** пароля (MD4 от UTF-16LE) либо иметь cleartext.
- bcrypt — односторонняя соль-функция; из неё NT-hash/cleartext не восстановить.

Следовательно, с bcrypt в `auth_credentials.password_hash` FreeRADIUS **не сможет**
аутентифицировать EAP-MSCHAPv2. Конфиг auth-бэкенда заблокирован до решения.

## Варианты
- **A. Хранить NT-Password (MD4)** в `auth_credentials` для `rlm_sql`+`mschap`.
  NT-hash несолёный и слабый по своей природе, НО: пароль генерим мы, он случайный
  (32 символа, высокая энтропия), отдаётся один раз в .mobileconfig; БД под FDE/at-rest
  шифрованием. Сохраняет flow README и простоту MMVP.
- **B. Перейти на EAP-TLS** (клиентские сертификаты): пароля нет вовсе, узлу нужен лишь
  CA. Снимает проблему, но требует PKI/CRL и пересмотра ADR-0005; аккаунтинг/ревокация —
  отдельно. Это была альтернатива из ADR-0005.
- **C. Cleartext-Password** — не рассматриваем (хуже NT-hash без выгод).

## Рекомендация
**A** на MMVP: минимальное изменение (колонка хранит NT-hash вместо bcrypt; переименовать
в миграции), сохраняет выбранный EAP-MSCHAPv2 и flow Plati. К **B** имеет смысл вернуться,
если потребуется более сильная серверная стойкость к утечке БД.

## Следствия (после решения)
- Миграция: семантика/имя колонки `password_hash` (NT-hash) — при варианте A.
- README §2.7 шаг 5 (bcrypt) корректируется под выбранный вариант.
- Только после этого пишется FreeRADIUS-роль control-plane (rlm_sql + mschap).
